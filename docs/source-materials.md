# 原材料处理与评估记录

最后更新：2026-06-22

## 处理原则

`原材料/` 用于临时保存原始访谈、ChatGPT 导出对话、外部评估、附件说明和未筛选素材。这类材料可能包含个人账号链接、临时判断、附件上下文或未经验证的建议，因此默认不进入 Git。

进入版本库的只应是提炼后的结论，例如：

- 项目章程中的原则和边界
- 架构计划中的已采纳技术路线
- 测试维护文档中的质量规则
- README 中面向用户的稳定表达

处理流程：

1. 原文放入 `原材料/`。
2. 先评估可采纳结论、风险和不采纳内容。
3. 只把稳定、可执行、符合项目定位的内容提炼进 `docs/`。
4. 不提交原始对话、外部附件、私有链接和未筛选长文。
5. 如确需公开原材料，应先移除个人链接、附件痕迹和不可验证陈述。

## 2026-06-18 原始对话评估

原材料：

- `原材料/20260618_原始对话_ChatGPT-架构与工程评估.md`

总体判断：

这份对话对项目定位有参考价值，但不应直接作为产品文案或开发计划。它的主要价值是从外部视角验证了三个方向：当前 Windows 本地优先技术栈合理，模块化已经接近 PowerShell WinForms 形态的上限，虚化穿透模式是项目最有差异化的产品能力。

可采纳结论：

- 当前项目本质是 Windows 本地优先的轻量桌面执行工具，不应短期迁向 Electron、云同步或复杂后台服务。
- 现有模块化方向正确，但 PowerShell 项目天然存在隐式耦合风险，后续应继续把散落的 `$script:` 状态收敛到 `$App.Ui`、`$App.Window`、`$App.Timer` 等显式容器。
- `Task`、`Today`、`Archive` 和 `PomodoroEngine` 的抽象是合理的，值得保持简单稳定。
- `WatermarkMode` 不只是“透明模式”，而是低打扰、可穿透、持续存在的执行状态层；后续对外表达应强调“减少注意力切换”和“维持执行状态”，而不是堆功能。
- 质量路线应继续补强：自测、状态流转检查、模块边界检查和发布清单比新增大功能更重要。

谨慎采纳：

- “行为操作系统”“注意力 OS”等说法适合作为内部探索概念，不适合直接进入当前 README 或主界面文案，容易显得过度承诺。
- 对产品评分、商业潜力评分只能作为讨论材料，不能作为路线决策依据。
- 注意力建模、行为预测、AI 建议、遥测和插件体系均不进入近期计划。
- 中期技术演进可以考虑 C#/.NET 或 WPF，但前提是 UI 复杂度或可维护性压力真实出现；当前仍以 PowerShell 5.1 + WinForms 为主。

本次落地：

- `.gitignore` 已加入 `原材料/` 和 `raw-materials/`，避免原始对话进入版本库。
- 本文档作为可提交的提炼记录，保留评估结论和处理规则。
- 暂不因该原始对话修改功能代码。
## 2026-06-21 Clean/Event-driven 架构方案评估

原材料：

- 用户贴入的 ChatGPT “当前架构 -> Clean Architecture + Event-driven 重构对照图”。

总体判断：

该方案对当前风险的判断基本成立：项目已经有模块化基础，但 service/workflow 层容易变重，UI、业务编排、运行期副作用和基础设施之间仍有耦合残留。尤其是虚化和翻译必须完全分开，这一点已提升为架构硬边界。

可采纳结论：

- 采用分层方向：表现层、应用工作流、领域规则、运行期能力、基础设施适配器和质量闸门必须分清。
- 采用横切隔离：虚化、翻译、音频、窗口状态和通知不应散落在业务流程里。
- 采用少量同步事件通知，用于设置变更、番茄完成、翻译完成/失败等横切响应。
- 将 UI 直接组合复杂业务动作的路径逐步收敛到 `TaskWorkflow`、`PomodoroWorkflow`、`SettingsWorkflow`、`TranslationWorkflow`。

谨慎或暂不采纳：

- 暂不引入完整 Command Bus；当前 WinForms 事件直接调用明确 workflow 函数更容易排障。
- 暂不抽 `ITaskRepository`、`ISettingsRepository` 等 repository port；当前本地 JSON 存储固定，先用模块边界和自动检查控制依赖。
- 暂不强行把任务和番茄改成复杂 OO aggregate；优先用显式状态对象、纯函数状态迁移和自测覆盖规则。
- 不做一次性 Clean Architecture 重写；继续按文档中的架构切片逐步调整。

本次落地：

- `docs/architecture-clean-event-decision.md` 记录外部 Clean/Event-driven 方案的逐项采纳、降级和暂缓决策。
- `docs/architecture.md` 补充外部方案评估、目标依赖矩阵、目标架构总则、目标架构图和虚化/翻译硬边界。
- `docs/architecture-refactor-plan.md` 将已完成边界和下一步切片串成渐进式迁移计划，并补充翻译平台适配器中立化、虚化退出不拥有翻译生命周期等切片。
- `docs/watermark-translation-plan.md` 将生产口径从“虚化-翻译”调整为独立的“翻译增强”，并补充翻译平台适配器目标边界。
- `docs/window-state-lifecycle.md` 明确虚化/实化转换不拥有翻译生命周期。
